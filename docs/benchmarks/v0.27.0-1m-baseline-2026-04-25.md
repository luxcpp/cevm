# 1M-tx parallelism baseline (pre-v0.28 GPU work)

Apple M1 Max, 10-core CPU, 32-core GPU, 64GB unified memory.
Workload: simulated AMM swap (touches reserve0/reserve1, 19-byte bytecode),
maximum write-write contention (every tx writes the same two slots).

## Numbers (medians, raw — no CPU-Par re-execution data exposed yet)

| N         | CPU-Seq (ms) | CPU-Par (ms) | GPU-Met (ms) | par/seq | gpu/par |
|-----------|--------------|--------------|--------------|---------|---------|
|   100,000 |       203.47 |       180.48 |      1025.28 |   1.13× |   0.18× |
|   500,000 |      1065.64 |       902.83 |     13770.10 |   1.18× |   0.07× |
| 1,000,000 |          —   |          —   |      **OOM** |    —    |    —    |

GPU at N=1,000,000 crashes with `std::runtime_error: Metal buffer allocation
failed`. Root cause: kernel allocates 64 KiB of scratch per tx, so 1M × 64 KiB
= 64 GiB needed for one buffer — exceeds Metal's per-buffer limit and the
device's free pool.

## Why CPU-Par doesn't blow the doors off either at this scale

Block-STM has to validate every tx's reads-set against every concurrently
committed write-set. At maximum write-write contention every tx conflicts
with every other tx, so CPU-Par re-executes most of the block. Net is
1.13–1.18× over sequential — Block-STM's worst case.

This is exactly where GPU parallelism should shine: 1M txs that all execute
the same bytecode are perfect SIMD fodder, *if* the kernel doesn't
serialize through 64 KiB-per-tx scratch and a single host roundtrip.

## v0.28 fixes (in flight)

1. **Per-tx scratch sizing** via static bytecode analysis — drops per-tx
   footprint from 64 KiB to ~256 B for `amm_swap` (3-element stack, no
   memory, no storage cache, no output). 1M txs × 256 B = 256 MiB, fits
   comfortably.

2. **Async dispatch** with `addCompletedHandler:` and chained command
   buffers — eliminates the per-launch CPU↔GPU sync.

3. **SIMD-cooperative kernel variant** — 32 threads per threadgroup
   sharing hot threadgroup-local state, dramatically reducing the
   per-opcode interpreter overhead.
